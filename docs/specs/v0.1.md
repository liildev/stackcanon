# StackCanon v0.1

## Goal

Ship a CLI-first installer for existing JavaScript and TypeScript projects that:

1. Detects the current stack.
2. Resolves a validated framework pack.
3. Resolves a quality provider strategy.
4. Plans safe file creation and patching.
5. Generates AI-facing project context from a single source of truth.

## Supported Categories

- Frontend
- Backend
- Fullstack

## Supported Framework Packs

- Next.js
- Nuxt
- React + Vite
- Vue + Vite
- NestJS
- Express / Fastify
- TanStack Start
- TanStack Query guidance pack

## Supported Quality Providers

- Ultracite
- Biome
- Oxlint
- Skip

## Core Commands

- `stackcn init`
- `stackcn doctor`
- `stackcn generate`
- `stackcn sync`
- `stackcn add`
- `stackcn revert`
- `--json` on `doctor`, `init`, `add`, `generate`, `sync`, and `revert` for machine-readable output

## Default Policies

- Existing project required for framework packs.
- Unknown major versions stop by default.
- `--allow-compat` can opt into nearest supported major.
- `Biome` and `Oxlint` installed together is a hard conflict.
- Existing user-managed config files are never overwritten silently.
- Reserved StackCanon paths are `ai/*` and `.stackcn/manifest.json`.
- Reserved StackCanon paths stop install when they already exist as user-managed files.
- Marker-based ownership decides whether a file is `stackcn-managed` or `user-managed`.
- `stackcn` writes backups before applying changes.
- `--install` runs the detected package manager after apply and backs up the active lockfile first.
- `--vendor-init` is reserved for vendor-owned interactive setup flows and currently supports Ultracite only.
- `--run-ai-rulez` runs the derived `.ai-rulez/` output through a real `ai-rulez generate` subprocess.

## Local vs Online Responsibilities

- Local deterministic layer:
  - stack detection
  - pack resolution
  - conflict checks
  - ownership checks
  - safe patch planning
  - `ai/` generation
  - `AGENTS.md` and `CLAUDE.md` generation
  - docs sync state assessment from local `.stackcn/sources/index.json`
- Online or interactive layer:
  - package manager install during `--install`
  - vendor-owned setup during `--vendor-init`
  - docs fetch during `stackcn sync`
  - future pack refresh commands
- Core planning and file generation must stay deterministic without network access.
- Online commands must be explicit opt-ins and never hidden inside the default planning flow.
- Tests should prefer the local deterministic layer and stub the online layer.

## Generated Artifacts

- `package.json` additive patches for required devDependencies and scripts
- `.stackcn/manifest.json`
- `.stackcn/backups/...`
- `ai/...`
- `AGENTS.md`
- `CLAUDE.md`
- tool-specific AI outputs through the AI engine wrapper

## Ownership Model

- `generated by stackcn` marker means the file is safe to patch automatically.
- Missing marker means the file is user-managed and requires review.
- `stackcn generate` only rewrites stackcn-managed outputs.
- `package.json` is patched additively: missing dependencies and scripts are added, existing scripts are not overwritten.

## Source Registry

- Official source registry lives in `packages/packs/src/sources.ts`.
- Packs consume registry entries instead of hardcoding doc URLs inline.
- Registry stores:
  - product
  - URL
  - roles such as `config`, `data`, `ai`, `upgrade`
  - supported version range
  - `lastVerified`
- `stackcn sync` writes local source artifacts to `.stackcn/sources/raw/*`, `.stackcn/sources/normalized/*`, and `.stackcn/sources/index.json`.
- `doctor` reads the local source index and reports `docsSyncState` as `none`, `missing`, `partial`, `stale`, or `ready`.

## Tooling Registry

- Recommended tool versions, schema URLs, and vendor-init defaults live in `packages/tooling-registry/src/index.ts`.
- Framework and quality adapters must read versions from the tooling registry instead of hardcoding them inline.
- Updating supported baseline tooling should usually be a registry-only change plus targeted regression checks.
- `doctor` should use tooling registry metadata to explain why a baseline version is recommended.

## JSON Contracts

- Machine-readable CLI contracts live in `packages/contracts/src/index.ts`.
- `doctor --json`, `init --json`, `add --json`, `generate --json`, `sync --json`, and `revert --json` must stay compatible with these exported types and guards.
- `apps/web` should consume contract types and guards instead of re-describing CLI payloads locally.
- `apps/web` is an internal product prototype for inspecting shared CLI payloads, not the public marketing site.

## AI Engine Adapter

- `ai/` remains the canonical source of truth.
- `AGENTS.md`, `CLAUDE.md`, and `.ai-rulez/` are derived outputs.
- `.ai-rulez/` is generated from canonical `ai/` files so StackCanon can integrate with ai-rulez without making `.ai-rulez/` the primary authoring surface.
- `stackcn generate --target ai-rulez --run-ai-rulez` writes derived `.ai-rulez/` files first, then invokes `ai-rulez generate`.

## Integration Test Strategy

- Use fixture repos under `tests/fixtures` as the primary integration inputs.
- Run deterministic integration tests inside `packages/core/test` with the Node test runner.
- Cover:
  - `doctor` findings for readiness gaps and conflicts
  - `init` planning for validated and compatibility-mode packs
  - `apply` for user-managed and stackcn-managed config paths
  - `generate` for managed and user-managed outputs
  - `sync` for manifest-scoped official source fetch and normalization
- Stub online behavior:
  - inject a fake vendor-init runner instead of calling `npx ultracite init`
  - inject a fake dependency installer instead of calling `pnpm install` or equivalent
  - inject a fake ai-rulez runner instead of calling a real subprocess in deterministic tests
  - inject a fake source fetcher instead of hitting the network during docs sync tests
  - keep true network/package-manager verification separate from deterministic apply tests
- Reserve manual end-to-end checks for true interactive vendor flows and real dependency installation.
